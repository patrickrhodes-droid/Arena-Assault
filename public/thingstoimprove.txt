-- KNOWN RELIABILITY ISSUES --

segmentIntersectsExpandedBox uses obs.xMin/xMax/zMin/zMax but obstacles are stored
  as obs.min.x / obs.max.x / obs.min.z / obs.max.z. This means soldier/dog/skeleton
  detour pathfinding always returns false (no blocking obstacle found), so those
  enemies always go straight and get stuck behind walls instead of routing around them.
  Fix: change segmentIntersectsExpandedBox and getDetourDirection to use obs.min.x etc.

tickEnemies (server.js) is never called from startGameLoop. The server-side boss AI
  (attack, movement, escape jump) is dead code. All enemy AI runs on the host client.
  This is intentional per the architecture but means if the host disconnects mid-wave,
  all enemies freeze. Consider adding a note in server.js or eventually calling
  tickEnemies as a fallback when no client owns an enemy.

enemyMeleeAttempt server handler does not verify the sender is the host/owner of the
  enemy. Any connected client could theoretically send fake boss hits to damage other
  players. Low risk for LAN play but worth noting.

Boss escape jump now clears windupTmr and swingTmr on launch — ghost hits fixed.

-- PERFORMANCE IMPROVEMENTS IMPLEMENTED --
- Shadow map: PCFSoftShadowMap → PCFShadowMap (4-tap instead of 9-tap; same look at
  play distance, meaningful GPU cost reduction).
- Shadow map resolution: 2048×2048 → 1024×1024 (4× fewer shadow pixels to compute).
- HUD + status indicators: updated every 2nd frame instead of every frame.
- Minimap: redrawn every 3rd frame instead of every frame (canvas 2D ops are expensive).
- Bullet loop: eliminated per-bullet position.clone() each frame; previous position is
  now stored directly on the bullet. Eliminated per-bullet×per-enemy Vector3 allocation
  by reusing module-level _enemyCenterVec/_hitPosVec cached vectors.

-- PERFORMANCE IMPROVEMENTS NEEDING COMPROMISE OR LARGER EFFORT --
- Geometry instancing: identical props (barrels, crates, bollards) are separate draw
  calls. THREE.InstancedMesh would reduce draw calls by 30-50% on prop-heavy maps.
  Requires refactoring loadProp to collect instances before rendering.
- Shared material caching: each map creates new MeshStandardMaterial objects; caching
  across rebuilds would cut shader compilation pauses on map change.
- Occlusion culling: Three.js has no per-object occlusion culling. For Blacksite with
  its many interior walls, a BVH-based system (three-mesh-bvh) could skip rendering
  objects in other corridors. Significant third-party integration needed.
- Web Workers for AI: enemy AI runs on the main thread and spikes on large waves.
  Moving to a Worker would keep the render thread smooth but requires a complex state-
  sharing refactor.
- Texture atlassing: each GLB prop loads its own textures; atlassing would reduce GPU
  texture-swap overhead but needs an asset pipeline change.
- FXAA instead of MSAA: the renderer uses antialias:true (MSAA), which is expensive.
  Switching to FXAA postprocessing would be faster but softer-looking at edges.
